# 先验证,再教学:面向数字硬件设计教育的执行门控可信 LLM 辅导框架与系统

## 摘要

大语言模型(LLM)导师已经服务于数十万学生,但对已上线系统的多项审计一致表明:它们给出的反馈中,有相当一部分存在技术错误;修复类帮助是最薄弱的类别;未经监督的回答只有少数经得起教学团队审核;面对错误的学生解答,模型还会系统性地"把错的说成对的"。错误反馈和正确反馈一样流畅、一样自信,学生根本无从分辨;而提示词层面的防护措施,已被证明会随使用规模扩大不断失效。本文的主张是:凡是学生作业可以由机器检验的学科,反馈可靠性就应当是一个工程问题,而不该赌模型能力的运气。本文有三项贡献:(1)提出"验证门控辅导"设计框架——四条原则,保证任何 AI 生成的评判、讲解、修复代码或练习步骤,不经执行验证器确认,一律不得送达学生;(2)实现参考系统 TrustGrade,面向数字硬件设计(Verilog)教学,由分层评估骨架(编译检查、性质测试平台、差分测试)、反馈门和脚手架验证器组成;(3)在 221 份提交、金标由构造法建立的语料上完成技术评测。结果显示:骨架为 95.9% 的提交给出了验证器背书的评判(95% CI [92.4, 97.8]),其准确率达 99.5% [97.4, 99.9];在三种对抗性的"导师出错"场景下,反馈门拦截了全部有害反馈(76/76,[95.2, 100]),单次检查约 71 毫秒;交付的 36 个脚手架步骤全部可运行,12 个最终步骤全部可证满足作业要求。我们进一步复现了文献中的五种门控范式,六个条件全部跑在同一批 222 条语料上,其中 85 份坏修复含 9 份绕开性质测试的规避例:完整执行门与前沿 LLM 审查门在准确率上统计不可区分(McNemar 精确 p = .73)而成本近零;它显著优于纯测试门(p = .039),差异集中在性质规避类(拦下 8/9 对 0/9);普通模型审查器错扣了 51% 的合格反馈;对已发表最强门——模拟学生验证——的忠实复现表明其对代码工件全盲,85 份有害修复交付了 79 份。上述结论在另一套独立编写、横跨十四类设计的 130 题语料(664 份提交,完整公开)上原样复现;在入门 Python(MBPP)上的跨域检验则既把机制推广到硬件之外,也划出了它的边界:执行门以近零成本保持可靠,而普通 LLM 审查器误拒了 43% 的正确解——不过前沿审查器在这类教科书任务上几乎追平执行,说明门的检测优势最大处在 HDL 这类专门、训练欠代表的领域。近期评测呼吁 LLM 导师采用"可靠接地组件管诊断、LLM 管对话"的混合架构——执行门控正是这类"接地诊断组件"在可执行学科的落地方案。文末讨论了这一路线对教育 AI 公平性与可信性的意义。

**关键词:** 智能辅导系统;大语言模型;自动评估;形成性反馈;可信人工智能;硬件设计教育

## 1. 引言

生成式 AI 辅导已经走出实验室,成为日常教学基础设施的一部分。哈佛 CS50 的 AI 助手一学年回答了二十多万学生的约一千万次提问(Liu et al., 2025);CodeAid 等课堂助手在大型编程课上运行了整个学期(Kazemitabaar et al., 2024);商用 AI 导师正在被成批采购。它的教学价值是实在的——即时、个性化、随叫随到的形成性反馈,历来是影响学习效果最有力的因素之一(Hattie & Timperley, 2007)。但恰恰是这些证明了价值的部署,也把一个结构性缺陷量化了出来:**LLM 的反馈并不可靠,而且错的时候和对的时候一样自信。**

已有文献在这一点上结论高度一致。CodeAid 的课堂部署审计发现,相当比例的回复存在技术错误,其中修复类帮助明显更差(Kazemitabaar et al., 2024)。Azaiz 等(2024)对 GPT-4 就真实入门课作业生成的反馈做了独立检查:完全正确且完整的仅约半数,还有一部分前后自相矛盾,作者据此建议不要无监督部署。CS50 的论坛上,AI 回答只有少数获得助教背书;尽管有明确的"不许直接给代码"指令,含代码的回复仍随规模扩大持续出现——护栏被一点点"稀释"掉了(Liu et al., 2024, 2025)。最新的证据来自 Yasir 等(2026b):他们对多个 LLM 逻辑辅导代理的大规模评测发现,模型会系统性地认可错误解答、错拒合法但写法不常见的解答;换哪个模型几乎决定了反馈质量的全部差异,把完整标准答案喂给模型也无济于事。作者据此呼吁混合架构:**诊断交给 KG 接地的组件,对话与脚手架交给 LLM。**

在教育场景里,这些错误率的分量比在一般软件辅助里重得多。一条误诊错误、认可坏解、或者把代码"修"出新 bug 的反馈,不只是没帮上忙——它顶着导师的权威,把误概念种进了学生脑子里。形成性反馈研究早就证明,学生对说得笃定的反馈几乎照单全收(Shute, 2008);对 AI 导师,自动化偏见还会进一步放大这种盲从。现有系统的应对办法各有死穴:提示词约束会在规模化中失效(Liu et al., 2025);人工审核队列(CS50 的做法)等于把 AI 本来要省掉的专家人力又搭了回去;用**另一个 LLM** 来把关(Phung et al., 2023, 2024),则把审查模型自身的不可靠继承了下来——这个担忧有充分依据:大量研究表明,LLM 在没有外部信号时无法可靠地检验自己的输出(Gou et al., 2024; Huang et al., 2024; Kamoi et al., 2024);LLM 判官在以正确性为标准的对比上接近乱猜(Tan et al., 2025);让模型静态识别"看着对、其实错"的代码同样不可靠(Gu et al., 2024)。

但有一类学科,这个两难在其中根本不存在:**作业成果本身就能运行的学科。** 学生写的 Verilog 设计可以编译、可以放到断言了教学目标行为不变量的测试平台上仿真、可以和参考实现做差分比对——整个过程是确定性的,毫秒级完成,边际成本几乎为零。数字硬件设计同时也是教学资源严重跟不上需求的学科:全球集成电路设计人才紧缺,半导体人才培养已上升为多国的国家战略,而智能教学工具却近乎空白。目前唯一公开发表的 RTL 设计 AI 导师,用 LLM 自动生成的量规给学生代码打分,就现有文献看没有任何验证把关(RTL-SMARTIE, GLSVLSI 2026)。我们系统检索了 2023–2026 年的相关文献,**没有发现任何领域存在"每条反馈送出前都经过执行验证"的辅导系统**(§2.5)。

因此,本文要回答的核心问题是:**如何架构一个基于 LLM 的辅导系统,使送达学生的内容之可靠性,不再取决于 LLM 本身可靠不可靠?** 我们的回答分三层,沿用本刊设计类研究已确立的"框架—实例—评测"结构(Pozdniakov et al., 2024):

1. **设计框架**(§3):验证门控辅导的四条原则——LLM 只当讲解者,执行验证器才是裁决者;凡交付内容,要么已验证,要么明确标注"未验证";门只能放行或扣下,不许改写;评估接受一切满足要求的解法,而不是只认一个标准答案。
2. **参考实现** TrustGrade(§4):三层执行评估骨架;逐条核查 LLM 辅导消息的反馈门;为 LLM 生成的多步练习做认证的脚手架验证器;把验证器信号转成形成性反馈的诊断层。
3. **技术评测**(§5):221 份提交的构造法金标语料,外加 222 条对抗性反馈语料;复现文献中的五种门控范式、在同一批条目上跑齐六个条件,全部结论配精确推断统计;随后在另一套独立编写、横跨十四类设计的 130 题语料(664 份提交;§5.5)上复现,并在入门 Python 上检验跨域迁移(§5.6)。

我们也把话说在前面,明确本文主张的边界:在本文各套语料上,前沿 LLM 审查器的准确率与执行门打平(两套 HDL 语料上 McNemar 精确 p = .73 与 p = .62;Python 上 p = 1)。执行门胜在成本、确定性和有契约背书的保证,而不是检出率更高;而正如跨域检验所点明的(§5.6),这一检出优势最大处在 HDL 这类专门、训练欠代表的领域,在教科书式 Python 上则很小。本评测是技术性、对抗性的,不是课堂实验(§7)。这套方法也只适用于成果可被机器检验的学科——编程、硬件设计、形式化数学是,议论文写作不是。

后文安排:第 2 节综述 LLM 辅导的可靠性问题、反馈门控的已有做法、自我修正的相关结论,以及硬件设计教育中的 AI,末尾归纳研究缺口;第 3 节建立设计框架并从反馈理论推导评测指标;第 4 节介绍 TrustGrade;第 5 节报告评测;第 6 节讨论;第 7 节交代局限;第 8 节总结。

## 2. 相关工作

### 2.1 LLM 辅导系统的反馈到底有多可靠

已部署的编程教学 LLM 助手,构成本文的经验背景。CodeAid(Kazemitabaar et al., 2024)带着精心设计的教学约束提示词,在一门大型 C 语言课程上运行了整个学期;对抽样使用记录的人工审计显示,相当比例的回复技术上不正确,修复类帮助尤其薄弱;学期中途的更新(提示词精修连同模型升级)有改善,但没能消除错误。CodeHelp(Liffiton et al., 2023)提出了三段式提示"护栏",但只报告了学生的主观感受,没做正确性审计。CS50.ai(Liu et al., 2024, 2025)是记录最全的最大规模部署,其审计结果两头都不乐观:准确性上,只有少数论坛回答获得助教背书;护栏耐久性上,"禁止给代码"的指令在规模化使用中被持续"稀释"。Azaiz 等(2024)给出了无监督 GPT-4 反馈质量最干净的独立测量:完全正确且完整的仅约半数。编程之外,Yasir 等(2026a, 2026b)在取自真实学生交互日志的基准上评测了多个 LLM 逻辑证明辅导代理:模型确认标准答案近乎满分,却恰恰在最需要反馈的地方失手——错解被认可、合法的非常规解被错拒,且各模型差异悬殊。他们还试了加一个可以改写反馈的 LLM 判官:弱导师被拉高,强导师反而被拉低——判官把本来对的反馈改得空泛了,作者称之为"验证过度规范化"(Yasir et al., 2026a)。这批文献共同指向本文要解决的问题:失效是架构层面的,靠提示词修不好。

### 2.2 在交付前给 LLM 反馈把关的已有做法

另一条规模较小的工作线,是在 LLM 和学生之间加一道校验。PyFiXV(Phung et al., 2023)让第二个模型根据解释复现修复,以此校验 Python 语法错误的讲解——精度靠丢弃相当一部分反馈换来。GPT4Hints-GPT3.5Val(Phung et al., 2024)把执行得到的失败测例信息加进生成提示,再用 GPT-3.5 扮演"模拟学生"为每条提示把关,达到已发表门控中最高的精度,但仍要扣下不小比例的反馈——而且它本身还是一个 **LLM 门**。CodeTailor(Hou et al., 2024)在 Python 里用"通过全部内置单元测试+贴近度检查"把关一种制品——为 Parsons 拼图题生成的个性化解代码——未过关的自动回退到已验证的常见解,因此交付的拼图题始终基于正确代码。LeanTutor 用 Lean 证明助手对数学反馈做形式验证,但它把自然语言翻译成 Lean 的入口环节本身没有验证、错误率不低——信任问题只是挪了个地方。检索增强和知识图谱可以减少反馈里的幻觉,但给不了交付时的保证(Nongkhai et al., 2026)。Daheim 等(2024)训练验证器去**引导**反馈生成(软引导,不硬拦);"Dean of LLM Tutors"(2025)用 LLM 评估器筛查导师反馈。与上述工作相比,本文有三点不同:我们的门是**确定性的执行验证器**,不是模型;它覆盖**所有**送达学生的制品(评判、讲解、修复、练习步骤),不是单一品类;它落在一个"跑一遍测试"本来就是该学科正确性标准的领域——硬件设计。

### 2.3 为什么不能让 LLM 给自己把关

自我修正方向的文献解释了 LLM 门为什么靠不住。CRITIC(Gou et al., 2024)的结论是:完全依赖没有外部反馈的自我修正,"收益甚微,甚至越改越差";没有解释器反馈的程序修复"能力有限且不稳定"。Huang 等(2024)证明,一旦去掉暗中使用标准答案的停止条件,内在自我修正反而**降低**推理准确率。Kamoi 等(2024)综述后认定:文献里报告的自我修正成功案例,几乎都暗中用了标准答案。具体到代码,Gu 等(2024)测得模型识别"看着对、其实错"程序并不可靠,多个模型接近乱猜;Chen 等(2025)发现自己生成的测试会带偏自我调试。LLM 判官方向的结论与此呼应:即便前沿判官,在以正确性为金标的对比上也接近乱猜(Tan et al., 2025),系统性偏差已被逐类编目(Ye et al., 2025)。这些证据合起来,给出了本文门控条件对比(§5.4)所要检验的预测:**审查式的门,质量随审查模型起伏;执行式的门,不随。**

### 2.4 硬件设计教育中的 AI

硬件描述语言(HDL)教学的非 AI 工具相当成熟——HDLBits 让仿真判题练习普及开来,VPLAS(2025)给出了带规则式判分的引导学习系统——AI 工具却很薄。在代码生成一侧,VerilogEval(Liu et al., 2023)、RTLLM(Lu et al., 2024)等基准都用测试平台执行来评测 LLM 的 RTL 生成能力——即便前沿模型也远谈不上可靠;拿测试平台反馈进训练闭环,更是代码生成的标准做法(如 VeriReason, 2026)。而面向教学的 AI 刚刚起步:RTL-SMARTIE(GLSVLSI 2026)是一个多智能体 LLM 导师,评估模式用 LLM 生成的量规打分,公开材料中看不到反馈路径上有任何执行或形式验证。这正是促成本文的那个反差:**执行验证在 HDL 的工程实践里、在 LLM 的基准评测里无处不在,唯独在 HDL 的教学里缺席。**

### 2.5 研究缺口与本文贡献

综合 §2.1–2.4:(i)未把关的 LLM 辅导反馈,其错误率与护栏失效程度,已达到反馈理论认定学生会照单全收的量级(Kazemitabaar et al., 2024; Azaiz et al., 2024; Liu et al., 2025; Yasir et al., 2026b);(ii)现有的门,提示级的会失效,人工的没法扩展,LLM 式的继承模型的不可靠(§2.3 的预测,本文 §5.5 对普通模型证实);(iii)执行门控只在本领域之外、且只针对单一制品出现过;(iv)HDL 教育中——据我们所知,任何领域中——都还没有对**每一条**送达学生的反馈做交付前执行验证的公开系统。本文的框架、参考系统与评测,就是为填补这一缺口,并明确交代这套方法的保证到哪里为止。

## 3. 问题的形式化与验证门控辅导框架

本节先形式化系统要处理的对象(§3.1),再点明未加门控的 LLM 辅导留下的"交付问题"(§3.2);随后把四条设计原则写成形式化要求(§3.3),并把框架的保证写成命题、逐一给出证明(§3.4);§3.5 把这套形式机制对接到形成性反馈理论,得到 §5 所度量的指标。全节用语料中的一个真实题目——4 路轮询仲裁器——作贯穿示例,让每个定义都落在真实的课程数据上。

### 3.1 基本设定与记号

**定义 1(学习目标)。** 一个学习目标是五元组 `o = (σ, I, r, Φ, H)`:`σ` 是自然语言题面;`I` 是模块接口(端口名、方向、位宽);`r` 是参考实现;`Φ = {φ₁,…,φₖ}` 是一组有限的**可执行行为性质**,落地为一个自检测试平台——驱动接口、对波形逐条断言每个 `φⱼ`;`H` 是**差分测试激励**,即针对 `I` 输出的激励发生器和观测窗口。

*贯穿示例。* 仲裁器题目中,`σ` 要求 one-hot 授权在四个请求方之间公平轮转;`I = (clk, rst_n, req[3:0], grant[3:0])`;`Φ` 有三条——`φ₁`(互斥):每拍 `onehot0(grant)`;`φ₂`(不许空授权):`grant ∧ ¬req = 0`;`φ₃`(有界等待):持续拉高的 `req[i]` 必须在 `N+2` 拍内拿到 `grant[i]`——由测试平台在随机激励和保持请求两种场景下断言。本文语料共 37 个题目(协议/流控 12、时序 12、跨时钟域 7、组合 6),每个测试平台的断言点中位数为 3(1–11 不等)。

记 `S` 为提交空间(声明了接口 `I` 的 Verilog 源码)。金标准以**理想契约**定义:令 `Φ*` 为题面 `σ` 蕴含的全部行为要求(一般是无穷集),提交 `s ∈ S` **达标**当且仅当 `s ⊨ Φ*`。可执行的 `Φ` 只是它的有限欠近似,`Φ ⊆ Φ*`。这个包含关系——测试平台只断言题面确实要求的东西——是出题人必须遵守的约束,我们称之为**契约有效性**;下文所有可靠性结论都只依赖这一条假设。记号:`s ⊨ Φ` 表示性质测试平台在 `s` 上全部通过;`compiles(s)` 表示编译成功;`s ≡_H r` 表示在 `H` 下 `s` 与 `r` 的观测序列完全一致。

**定义 2(评分器与两类错误)。** 评分器是函数 `V : S → {ACC, REJ}`(或下文的三值版本)。对照金标:**假阴性率** `FNR = P(V(s) = REJ | s 达标)`,度量*不公平*——合格作业被错判;**假阳性率** `FPR = P(V(s) = ACC | s 不达标)`,度量*不安全*——错误作业被放过。§5.4 会同时估计这两项;合格类中特意放入了结构迥异的多种实现,让 FNR 能真正量出评分器对"合法的不同写法"的惩罚。

### 3.2 交付问题

辅导流水线送达学生的是各种*制品*:评判、讲解、修复代码、练习步骤。LLM 导师是一个随机生成器 `g : (σ, s, 上下文) → m`,输出消息 `m = (v_t, e, f)`——它宣称的评判、一段讲解、一份修正代码。§2.1 中那些被审计的系统,实行的都是*原样交付*:`g` 说什么,学生就收到什么。于是,有害制品——判错的结论、修不好的代码、做不下去的练习步骤——落到学生手里的概率,就等于 `g` 自己的出错率:随模型而变、交付时无从得知、且在最要紧的类别上被实测证明相当可观(Kazemitabaar et al., 2024; Yasir et al., 2026b)。

所谓**交付问题**,就是设计一个交付策略 `D`,使有害交付的概率只由**契约** `(Φ, H)` 决定——契约是看得见、改得动的工程制品——而与 `g` 无关。此后,LLM 的水平只影响学生多久能收到一次丰富的反馈(覆盖率),不再影响收到的内容对不对。

### 3.3 四条设计原则的形式化表述

**P1(裁决权归验证器)。** 一切关于制品正确性的判断——`s` 达标吗?`f` 达标吗?第 `i` 步做得下去吗?——都由执行验证器给出;`g` 永远不是正式裁决者。*依据:* Yasir 等(2026b)发现的架构性失效——反馈质量的差异几乎全部由换哪个模型决定——意味着任何依赖 `g` 的裁决都是在碰运气;P1 干脆把 `g` 从可信基座中整个移出去。

**P2(不许输出未经验证的笃定)。** 交付策略的值域只有两种:{已验证的内容} ∪ {明确标注的不确定}。形式上,每个交付制品 `a` 都带状态 `st(a) ∈ {VER, UNC}`,且只有存在执行检查为其背书时才允许 `st(a) = VER`。最危险的那个象限——顶着导师权威的未验证内容,也就是 Shute(2008)效度分析里点名的"有害反馈"——在机制上就到不了学生面前。

**P3(只放行或扣下,绝不改写)。** 门的动作空间是 `{deliver(m), withhold(m)}`,没有"改写"这个动作。这是*门*与 Yasir 等(2026a)那个 LLM *判官*的本质区别:判官的伤害路径——把本来对的反馈改得抽象空泛——需要的正是一个我们的门不具备的动作。

**P4(接受整个解空间)。** 评估谓词只针对题面蕴含的行为(即是否满足 `Φ*`)来写,绝不针对与 `r` 的相似度来写。于是,凡满足要求的实现——包括出题人没想到的路数——一律接得住。这正是评估效度对发散性解法的要求(LLM 导师对合法非常规解的高错拒率;Yasir et al., 2026b)。

### 3.4 判定、门与保证

**定义 3(分层骨架判定)。** 骨架判定函数 `V_B : S → {ACC, REJ, UNC}`:

```
V_B(s) = REJ    若 ¬compiles(s) 或 s ⊭ Φ        (L0、L1)
       = ACC    若 s ⊨ Φ 且 s ≡_H r             (L2 一致)
       = UNC    若 s ⊨ Φ 且 s ≢_H r             (L2 不一致)
```

仿真不终止由看门狗超时截断,记为 `s ⊭ Φ`(挂死违反一切活性类性质的有界响应含义)。三值设计不可省:只用 L1,能给出 P4 要的自由,却会漏掉绕开不变量的 bug;只用 L2,就退化成违反 P4 的"对答案"式评分;而两层*意见相左的那个区域,恰恰是系统不该擅自拍板的地方*。

**命题 1(拒绝的可靠性)。** *在契约有效性成立(`Φ ⊆ Φ*`,且测试激励处于题面规定的工作条件内)的前提下,`V_B(s) = REJ ⇒ s` 不达标。*
*证明。* `V_B(s) = REJ` 有两种来源。其一 `¬compiles(s)`:`s` 在接口 `I` 上未实现任何行为,平凡不达标。其二,测试平台展示出一条使某 `φⱼ ∈ Φ` 失败的波形:因 `φⱼ ∈ Φ*`、且该波形处于契约的工作条件内,故 `s ⊭ Φ*`。∎

**命题 2(接受风险的刻画)。** *若 `V_B(s) = ACC` 而 `s` 实际不达标,则 `s` 偏离契约的每一处行为,既不被 `Φ` 的测试平台波形触发,也不落入 `H` 的观测窗口。*
*证明。* `ACC` 要求 `s ⊨ Φ` 且 `s ≡_H r`。设 `s` 违反某条 `φ ∈ Φ* \ Φ`,则按"违反"的定义存在一条区分波形:若该波形能被 `Φ` 的测试平台生成,则应有 `s ⊭ Φ`;若该偏离在 `H` 下可观测,则因 `r ⊨ Φ*` 应有 `s ≢_H r`。二者皆与 `ACC` 矛盾。∎

命题 2 划出了**残差类** `R(Φ, H)`——缺陷把两道检查同时躲过去的不达标提交。它就是这套架构的*全部*风险敞口:加性质、加宽 `H` 都会让它单调缩小;§5.4 的实测里,212 个可靠判定中它只出现了 1 次。这正是"可靠性是工程选择"这句话的形式内涵:**风险敞口是两个看得见的工件的可测性质,而不是某个模型的脾气。**

**定义 4(反馈门)。** 对关于提交 `s` 的导师消息 `m = (v_t, e, f)`:

```
G(m, s) = DELIVER(m)                 若 v_t = V_B(s) 且 V_B(f) = ACC   (C1 且 C2)
        = FALLBACK(V_B(s), δ(s))     否则
```

其中 `δ(s)` 是 §4.5 的诊断提取器。脚手架同理:步骤序列 `⟨(g_i, h_i, r_i, c_i)⟩ᵢ₌₁..ₙ`(目标、提示、步骤参考实现、检查点测试平台)只有在每对 `(r_i, c_i)` 都编译通过且检查点通过、并且最后一步满足真实契约 `r_n ⊨ Φ` 时,才允许交付。

**命题 3(不存在未经验证的权威)。** *在门 `G` 之下,凡到达学生的制品必属三类之一:(i)骨架给出的判定与诊断,由命题 1–2 背书;(ii)判定与骨架一致、且修复满足 `V_B(f) = ACC` 的 LLM 消息;(iii)明确标注 `UNC` 的升级项。特别地,若一份交付出去的修复居然不达标,它必然属于 `R(Φ, H)`。*
*证明。* 对定义 4 与图 8 的交付路径分情形讨论:LLM 内容抵达学生的唯一通道是 `DELIVER`,其守卫条件强制了 (ii);其余路径送出的都是骨架生成的内容 (i),或 `UNC` 升级 (iii)。末句即命题 2 在 `f` 上的实例。∎

**推论 1(与模型无关)。** *命题 3 的保证不含任何依赖生成器 `g` 的项。`g` 的水平只决定交付率 `P(C1 ∧ C2)`——学生多常收到丰富的 LLM 反馈、多常收到验证器的兜底反馈——而绝不决定收到的内容是否可靠。*

推论 1 是整个框架的中心主张,而且**可以被证伪**:§5.4(RQ4)的做法就是固定语料、只换门控机制——若交付内容的可靠性其实依赖模型,换门必然露馅。实测结果:执行门的拦截纹丝不动;审查式门则随审查模型起伏(前沿模型:与执行门相当;普通模型:5 次有害交付、62% 错拦)。

### 3.5 从反馈理论到可测指标

Hattie 与 Timperley(2007)提出,有效反馈须回答三问:*目标是什么?*、*我做得怎么样?*、*下一步怎么办?*。上面的形式机制把每一问都变成可测的要求:要*正确*回答"我做得怎么样",就要求判定准确——即 `V_B` 的覆盖率与准确率(RQ1);要*有信息量*地回答它,就要求 `δ(s)` 点名是哪条 `φⱼ` 被违反、给出观测证据——即诊断可操作性(随 RQ1 报告);要*安全*地回答"下一步怎么办",就要求交付的修复和步骤真的满足契约——即门的拦截率(RQ2)与脚手架有效性(RQ3)。Nicol 与 Macfarlane-Dick(2006)所说"向学生交付关于其学习的高质量信息",在可验证学科里由此不再是一句愿景,而是一个可以检查的后置条件——§5 就去检查它。

### 4.2 评估骨架

**接口约定。** 输入:提交 `s`、目标 `o`。输出:按定义 3 的 `(V_B(s), δ(s))`。失效处理:编译不过(附 L0 诊断)、断言违规(附第一条失败性质行)、仿真挂死(看门狗截断,归类为组合环/时钟故障)、接口不匹配(REJ,附端口差异)。

**算法 1——分层判定。**
```
function ASSESS(s, o = (σ, I, r, Φ, H)):
    if not COMPILE(s):            return (REJ, δ_compile)          # L0
    t ← SIMULATE(s, TB(Φ), 看门狗 W)                                # L1
    if t = TIMEOUT:               return (REJ, δ_hang)
    if 存在 φⱼ 在 t 中失败:        return (REJ, δ_invariant(φⱼ, t))
    (y_s, y_r) ← (OBS(s, H), OBS(r, H))                            # L2
    if y_s = y_r:                 return (ACC, "满足 Φ,且在 H 下与 r 一致")
    else:                         return (UNC, δ_divergence(y_s, y_r))
```

**贯穿示例。** 一份把指针更新写成 `base <= gidx`(而不是 `gidx + 1`)的仲裁器提交——这是个真实 bug,效果是刚被服务的请求方继续霸占最高优先级——能通过编译(L0),也满足 `φ₁、φ₂`;但在保持请求 `req = 0101` 时,2 号请求方被饿死。L1 判 `REJ`,诊断信息原样取自语料:`δ = "VIOLATION starvation: req[2] unserved for 5 cycles (req=0101 …)"`。反过来,掩码式仲裁器——结构上和旋转指针参考实现完全不同——三条性质全过,授权时序在 `H` 下又与参考一致,判 `ACC`;计数扫描式变体的授权时序合法地不同,落入 `UNC`,而不是被错杀——这正是三值语义的用意。以上三个制品全部取自评测语料,没有一个是为写论文编出来的。

**各层耗时与产出。** 语料实测(Apple M4):中位数 L0 约 9 ms、L1 约 37 ms(只有挂死时才会顶到 W = 3 s 的看门狗)、L2 约 28 ms(含参考实现那一遍);端到端中位约 75 ms/次(文中另一处的约 71 ms,是单独测得的门检中位数)。这个速度足以支撑"边写边判"。评测中各层产出:L0 拒 7 份、L1 拒 68 份、L2 收 137 份、升级 9 份(§5.4)。

### 4.3 反馈门

**接口约定。** 输入:导师消息 `m = (v_t, e, f)`、提交 `s`、目标 `o`。输出:`DELIVER(m)` 或 `FALLBACK(V_B(s), δ(s))`。门对讲解 `e` 一个字也不改(P3)。

**算法 2——门。**
```
function GATE(m = (v_t, e, f), s, o):
    (v, d) ← ASSESS(s, o)                    # 骨架先裁决学生的提交
    if v_t ≠ v:                    return FALLBACK(v, d)          # C1
    (v_f, _) ← ASSESS(f, o)                  # 完整骨架检查建议修复
    if v_f ≠ ACC:                  return FALLBACK(v, d)          # C2
    return DELIVER(v, e, f, tag="verified")
```

有两个设计决策值得说明。第一,**C2 用完整骨架,而不只是性质测试平台**。在 §5.4(RQ2)的压测语料上,便宜的纯测试门恰好也够用——那里每份破坏修复都触犯了某条已断言的不变量;但评估语料说明了为什么保险还是要买:9/84 个 bug(10.7%)完全绕开了性质集(§5.4,RQ1),纯测试门继承的正是命题 2 中 `R(Φ, ∅)` 那份敞口。完整契约把它关上,代价是实测中多拦了一份合法修复。第二,**兜底反馈由验证器信号生成,而不是改写 `e`**:图 7 的 (b)、(d) 两格分别展示了拦截与放行两种情形下学生实际收到的东西。

![图7](figures/fig7_worked_example.png)

**图 7.** 取自语料的完整实例(轮询仲裁器)。(a) 优先级指针不轮转的学生提交;(b) 骨架的拒绝判定与语料原文的饥饿诊断;(c) LLM 导师起草的消息;(d) 门检 C1/C2 通过,验证过的消息交付。若 C1 或 C2 不过,学生收到的是 (b)——有害反馈到不了学生手里。

### 4.4 验证脚手架

**接口约定。** 输入:目标 `o`、分解深度 `n ∈ {3, 4}`。输出:通过验证的脚手架 `⟨(g_i, h_i, r_i, c_i)⟩`,或打回重新生成;凡交付的步骤,必满足下列 s1–s3。

**算法 3——脚手架验证。**
```
function VALIDATE-SCAFFOLD(⟨(g_i, h_i, r_i, c_i)⟩ᵢ₌₁..ₙ, o):
    for i in 1..n:
        if not COMPILE(r_i ∥ c_i):        return REGENERATE(i)     # s1
        if SIMULATE(r_i, c_i) ≠ PASS:     return REGENERATE(i)     # s2
    if ASSESS(r_n, o) 不为 ACC 类:        return REGENERATE(n)     # s3:r_n ⊨ Φ
    return DELIVER-ALL
```

生成器(带工具、在备课阶段运行的 LLM)为每一步产出:学生看的目标、以提问方式给出的苏格拉底式提示、这一步的参考实现,以及**检查点测试平台**——学生的半成品可以随时对着它自测,让形成性反馈出现在每一步,而不是只在最后。s3 这一条,是"验证过的脚手架"与"看着合理的脚手架"的分水岭:交付出去的步骤序列,可证一路做到满足契约的解。生成成本(生成—测试—修复循环消耗的 LLM 算力)按题目一次性发生,与学生人数无关;本评测中 36 个生成步骤全部通过验证、12 个终点全部满足契约(§5.4,RQ3)。那里也如实说明:原始通过率高,功劳在生成器的自测循环,不在一次性生成;交付时真正兜底的始终是验证器。

### 4.5 诊断提取

提取器 `δ` 把每次拒绝转成结构化诊断:`δ_compile(行号, 信息)`(语法/结构类)、`δ_invariant(φⱼ, 时刻, 观测值)`(行为类——哪条要求被违反、什么时刻、哪些信号值作证)、`δ_hang(类别)`(组合环/时钟不走)。有一个实现细节直接改变了主指标:取仿真日志里*第一条失败断言行*——上面带着 `φⱼ` 的名字和观测值——而不是取仿真器最后的汇总行,可操作诊断率从 9% 跳到 89%(§5.4,RQ1)。结构化诊断喂给两个下游:学生(作为"我做得怎么样"的回答),以及 LLM 讲解器——后者*可以*拿 `δ(s)` 当作已验证的讲解素材。(§5 的评测中,导师只凭题面和提交起草讲解;保证靠门,不靠提示词。)

### 4.6 实现说明

验证器平面约 1,400 行 Python,调度 Icarus Verilog;固定随机种子加看门狗,保证结果可复现。导师与脚手架生成器由前沿 LLM(Claude)扮演;§5.4(RQ4)的普通审查条件经公开 API 调用 DeepSeek-chat;全部提示词见附录 A。完整语料——题目、契约、金标生成器、各方判定——已归档,可供复现(见"数据可用性")。

## 5. TrustGrade 评估与门控辅导的评测

本评测有两个目的:其一,考察执行骨架所交付评估的内在质量——对合法替代解是否公平、对错误作业是否安全、诊断是否有信息量;其二,考察不同门控机制阻止有误辅导反馈到达学生的效果。评测数据为两套:221 份提交、37 个题目、金标由构造法建立的语料,以及 222 条、标签经预言机确认的对抗反馈语料。就评估目的,我们对比四个评分器——性质评分器、参照输出匹配、token 相似度、结构相似度——外加完整分层骨架。就门控目的,我们对比六个条件,每个条件操作化文献中的一种范式:(1)**不设门**条件:导师消息原样交付,即 CodeAid 与 Azaiz 等所审计部署的做法;(2)**模拟学生**条件:复现 PyFiXV/GPT4Hints 的验证范式——一个 GPT-3.5 档的学生模型(DeepSeek-chat,与其 GPT-3.5Val 同档)只拿到错误代码和导师讲解,尝试修复三次,至少两次修复通过性质测试平台才放行该消息;(3)**普通审查**条件:DeepSeek-chat 对照题面静态审查建议修复(Dean-of-LLM-Tutors 范式);(4)**前沿审查**条件:同上但换前沿审查器(Claude),跑两遍以测判定稳定性;(5)**纯测试**条件:只用作业的性质测试平台把关(CodeTailor 范式);(6)§4.3 的**完整执行门**。分析聚焦四点:(i)评分器的公平与安全;(ii)有害反馈的拦截率及误拦代价;(iii)交付脚手架的有效性;(iv)六个门控条件在交付内容正确性与成本上的差异。据此,评测回答以下研究问题:

- **RQ1:** 性质评分器与分层骨架交付的评估,在公平、安全与信息量上表现如何?与参照匹配、相似度评分器相比如何?
- **RQ2:** 执行式反馈门能在多大程度上拦下有误的辅导反馈?误拦与延迟的代价多大?
- **RQ3:** LLM 生成的脚手架经验证后,是否步步可执行、且可证通向正确解?
- **RQ4:** 在完全相同的条目上,六个门控条件在交付内容正确性、有害交付数与边际成本上有何差异?
- **RQ5:** 这些结论是否可推广——到另一套独立编写的 HDL 语料(§5.5),以及到另一个可执行学科即入门 Python(§5.6)?

### 5.1 语料与金标

评测评分器与门,金标须既不循环、又能规模化取得;我们因此构造之(图 6)。语料含 37 个题目,覆盖组合、时序、协议与跨时钟域设计,每题配题面、接口、参考实现与性质测试平台(断言点中位数 3,范围 1–11)。**合格类**(n = 137):各参考实现,加上仅当与参考的 token 级 alpha-等价被机器验证方可入库的语义保持变体,以及算法各异的已验证实现;137 份中 101 份为非同一替代解。**错误类**(n = 84):每题一个教师手写真实 bug,加变异注入缺陷,仅当独立差分预言机(2 种子 × 128 拍,区别于 L2 激励)确认可观测偏离方可入库;仅影响无约束行为的变异被排除。**对抗反馈语料**(n = 222):同一构造法的放大版——137 份合法"修复"(构造法证明的合格类,以导师建议修复的形式呈现),85 份破坏修复:36 份偷懒修复(学生错误代码原样退回充当"修复")加 49 份预言机确认的变异体(对每个参考实现施加全部适用的变异算子)。85 份破坏修复中有 9 份完全绕开性质测试平台(命题 2 的 `R(Φ, ∅)` 类),正是这一类给了语料区分顶部门控的能力,而『全部可被性质逮住』的缺陷总体不具备这种区分力。另有 58 份预言机下行为不变的变异体无法赋予可证标签,不进入主分析。条目匿名编号,各门控条件评判逐字节相同的制品。

### 5.2 流程与数据采集

RQ1:221 份提交交由四个评分器与分层骨架评判;各评分器读取完全相同的提交文件,逐份记录判定、裁决层与提取的诊断。RQ2:先由前沿 LLM 为 35 份教师编写的错误提交生成导师消息(判定、讲解、修正实现;提示词见附录 A.1);再以这些消息构造三种模拟失效模式——判定整体翻转、偷懒修复、变异修复——连同真实消息流,对门做压力测试。RQ3:以 §4.4 的带工具生成循环为 12 个代表性题目生成脚手架(36 步)并送验证器;逐步记录编译、检查点与终点契约结果。RQ4:六个门控条件在 222 条语料上执行:不设门条件全部交付;模拟学生条件每条发起 3 次修复尝试(666 次 DeepSeek 调用,温度 0.8),每次以执行判定成败;两个审查条件接收相同输入(题面、导师讲解、建议修复)与相同静态审查指令(附录 A.2),温度 0;纯测试与完整门条件直接执行制品。每个条件逐条记录判定、token 数与耗时。

### 5.3 数据分析

回答 RQ1:将各评分器判定与构造金标对照,落实定义 2——合格类上的假阴性率(对 101 份替代解单独报告)度量不公平,错误类上的假阳性率度量不安全。所有比例配 Wilson 得分 95% 置信区间,该区间在比例接近 0 或 1、样本量如本文时依然表现良好。评分器间误差率差异用双侧 Fisher 精确检验,适合本文这类极端单元格计数。回答 RQ2:各导师流的拦截率与放行率配 Wilson 区间;无需推断对比,因为不设门的对照按构造交付全部故障条目。回答 RQ3:步骤有效率与终点契约满足率报告为带 Wilson 区间的比例。回答 RQ4:各条件下的逐条正确性(有害消息被扣、合法消息被放)与完整执行门作对照,用不一致对上的精确双侧 McNemar 检验——这是小不一致数配对二元结局的恰当检验;报告 b(完整门对、对照条件错)、c(反向)与精确二项 p 值。前沿审查器两遍间的稳定性以原始一致率与 Cohen's κ 量化。预先指定的对比数少,不作多重性校正;全文报告精确 p 值,读者可自行调整。分析代码随工件归档。

### 5.4 结果

**RQ1:评估的公平、安全与诊断。** 性质评分器对 137 份合格提交零拒绝(FNR = 0.000,95% CI [0.000, 0.027]),对 101 份结构各异的替代解亦零拒绝([0.000, 0.037]),同时收下 84 份错误提交中的 9 份(FPR = 0.107 [0.057, 0.191])。Token 相似度与结构相似度评分器分别收下 82/84(0.976 [0.917, 0.993])与 81/84(0.964 [0.900, 0.988])份错误提交;Fisher 精确检验表明,两者的假阳性率与性质评分器差异显著(p = 2.0 × 10⁻³⁴ 与 p = 4.7 × 10⁻³³)。参照输出匹配错拒一份合法替代解(FNR = 0.007 [0.001, 0.040]),收下 11/84 份错误提交(0.131 [0.075, 0.219])。结果汇总于表 1,图 2 给出可视化。这一格局提示:在本语料上,相似度评分无论取何阈值都不安全;性质评分器的残余误差则局限于差分层所要关闭的性质覆盖缺口。

### 5.5 在独立公开语料上的复现与放大

单一、且取自课程的语料上的结果,有可能反映的是这套语料而非机制本身。为检验 RQ1 与 RQ4 的结论是否可推广,并给出一份无需(不可再分发的)课程材料即可运行的工件,我们从零另建了一套语料,并在其上重跑评测。这套**扩容语料**含 130 个全新题目,横跨十四类设计——组合逻辑、时序、状态机/协议、数据通路、算术(乘法器、除法器、超前进位/进位保存加法器、BCD、饱和运算)、存储(寄存器堆、RAM、FIFO、栈、内容可寻址匹配、环形缓冲)、总线/串行协议(UART、SPI、skid 缓冲、序列化、credit 流控)、DSP(FIR、MAC、积分器、comb 滤波、点积)、编码/纠错(奇偶、Hamming(7,4) 编解码、CRC-8、校验和)、位操作(找首/末置位、前导/后缀零计数、翻转、旋转)、进阶状态机、进阶时序(PWM、定时器、看门狗)、跨时钟域同步器,以及 ALU/控制数据通路。每题都配golden解、性质测试平台与一个真实缺陷,并且只有在机器验证golden过性质测试平台、缺陷被其判错之后才收录(130/130 全部通过;与主语料不同,这套语料完整公开)。随后以 §5.1 的同一构造流程扩为 131 题、**664 份提交**(402 份合法,其中 272 份是结构不同的替代解;262 份错误,其中 9 份绕开性质集)。

**评分器在大样本上原样复现。** 在 664 份提交上,性质评分器仍零误拒合法解(FNR = 0.000,95% CI [0.000, 0.009]),对 272 份替代解也零误拒([0.000, 0.014]);而 token 与结构相似度评分器分别收下 260/262(0.992 [0.972, 0.998])与 255/262(0.973 [0.945, 0.988])份错误提交——其假阳性率与性质评分器分别相差 p = 1.8 × 10⁻¹² 与 p = 9.7 × 10⁻¹³(Fisher 精确)。性质评分器的残差(收下 9/262)全部落在绕开性质集的常量自增变异上,恰为命题 2 的 `R(Φ, ∅)` 类。分层骨架给出的可靠裁决准确率为 654/656;作为交付门用于同一语料时,靠差分层拦下 262 份破坏修复中的 260 份(含 9 份规避例中的 7 份),误拦 1 份合法替代解(交付 401/402),交付 2 份有害修复——即绕开性质集与自动差分谐的残差 `R(Φ, H)`。同一语料上,纯测试门拦下 253/262、9 份规避例一份不中。

**六条件排序原样复现。** 在这套语料的一个 152 条子集(88 合法、64 破坏含 2 份性质规避)上跑齐 §5.4 全部六个门控条件——含两个付费 LLM 臂——重现了表 3 的排序(表 4):完整执行门零有害交付、两份规避例全中,与前沿审查统计不可区分(McNemar b = 3, c = 1, p = 0.62),而与普通审查(p = 7.6 × 10⁻⁵)、模拟学生范式(p = 6.1 × 10⁻²⁰)及不设门(p = 3.6 × 10⁻¹⁸)差异显著。普通审查再次过度封锁,88 份合法修复只放行 83 份,还漏放 13 份破坏修复;模拟学生因制品盲的构造,交付了 64 份破坏修复中的 48 份。这套语料是在框架定型之后才编写、横跨十四类设计、且完整公开——在其上的独立复现表明:§5.4 的结果是门控机制的性质,而非主语料的性质。

### 5.6 外部效度:Python 上的跨域检验

我们主张的机制——把对错判断交给执行——并不限于硬件。为探其可迁移的边界,并在一个 LLM 审查器本就强势的领域里诚实地把它放在对照中,我们在开源编程教学数据集 MBPP(Austin et al., 2021)上重跑了评分器/审查器对比。该集为 974 道入门 Python 题各配参考解与断言测试。以 §5.1 的构造流程,我们从 150 题建出 478 个候选——300 个正确(参考解加保义重排,均验证过测试)与 178 个错误(AST 变异,由测试判错确认)。Python 执行骨架在带看门狗的隔离子进程中对每个候选跑其测试,是性质测试平台的直接对应物。对照对象为 token 相似度,以及两个做静态正确性审查的 LLM 审查器——一个便宜模型(DeepSeek-chat)与一个前沿模型(Claude),各留出先推理再判定的空间,均不许执行。

有两点值得记,一个前提为它们定框。**前提**是:MBPP 上正确/错误的标签本就由执行骨架所跑的同一批测试定义,故此处骨架是预言机而非选手;有信息量的对照是几个静态方法相对于它的表现(表 5)。其一,**在常见 Python 上前沿 LLM 审查器几乎追平执行**:Claude 放行 300 份正确解中的 299 份、抓全 178 个错误——这些是前沿模型见得烂熟的教科书模式,它唯一的分歧是一份它主张确实缩进有误的重排。其二,**便宜 LLM 审查器严重过度拒绝**:即便开了推理,DeepSeek 也只放行 300 份正确解中的 171 份(误拒率 43%),同时抓住多数错误——与它在 HDL 语料上表现出的过度封锁如出一辙。token 相似度再度不堪用(178 个错误只抓 1 个)。

于是这组对照不是把结论普适化,而是把它磨得更锋利。在简单、训练中高度代表的任务上,前沿审查器就够用,执行相对它的优势很小;优势最大处在专门、训练中欠代表的领域(如 HDL)——§5.4 已表明,那里连前沿审查都会漏掉执行门能拦的性质规避修复。两种情形下,执行门都以近乎零的边际成本保持可靠;两种情形下,普通审查——唯一能扩展到资源不足场景的 LLM 选项——都最不安全。**只要存在可执行的预言机,这条架构守则跨域成立;而它红利的大小,取决于该领域在审查器训练中被代表得有多好。**

## 6. 讨论

### 6.1 主要发现

对照四个研究问题:执行骨架为 96% 的提交给出验证器背书的裁决,准确率 99.5%,同时一份合法替代解都没冤枉(RQ1);门把有害反馈的送达率从不设门时的必然 100% 压到三种对抗场景下的 0%,代价是约 3% 误拦和约 71 ms 延迟(RQ2);脚手架验证保证交付的每一步能编译、能通过、且整个序列可证走到正确解(RQ3);六条件同题对比表明,执行门站上了准确率的最前沿而成本近零,普通模型方案则两头同时失守(RQ4)。用 §3.5 的透镜看:这套系统把 Hattie 与 Timperley 的三个反馈之问变成了可检验的东西——"我做得怎么样"由验证过的判定和点名的不变量来答,"下一步怎么办"由被证明有效的修复和练习步骤来答。

### 6.2 与已有证据的关系

三点联系值得点明。第一,本文为 Yasir 等(2026b)的混合架构主张,在可验证学科给出了一个跑得起来的实例:他们报告的高错解认可率,恰好就是表 2 中被门从机制上消灭的那类失效。第二,Yasir 等(2026a)的负结果——"验证反而伤害强导师"——与本文不冲突,反倒印证了 P3:他们的伤害走的是 LLM 判官改写反馈那条路;我们的门只有"放行/扣下"两个动作,那条路根本不存在,而真实导师流的数据显示这样做的代价有限(35 条中误拦 1 条)。第三,普通模型审查臂把自我修正文献的警告(Gou et al., 2024; Gu et al., 2024)演成了实景:审查式的门继承审查者的水平,而在大家用得起的价位上,这个水平不够用——执行门则对水平不敏感。

### 6.3 对实践的启示

对可验证学科的系统开发者,结论可以浓缩成一条架构守则:**凡承载对错判断的内容,一律过执行检查;其余一切,交给 LLM。** 对办学机构,六条件对比揭示了一个公平性问题:LLM 审查式方案的安全水位跟着模型预算走,最付不起前沿模型的学校,反而会部署最不安全的导师;执行门把安全与开销脱钩——它跑在开源仿真器和普通电脑上。对评估设计者,P4 的结果说明契约式评分不只是更安全,还**更公平**:101 份替代解零冤枉,直接回应了"对答案"式判题器与 LLM 导师共有的那个毛病——对不走寻常路的解法有据可查的高错拒率(Yasir et al., 2026b)——而那恰恰是工程教育最想鼓励的解法。

### 6.4 边界

三处不越界的声明。其一,保证止于契约:性质集永远写不全(4.1% 的升级带和那一次判错,量出了当前预算的边界),加长差分激励、给 CDC 类 bug 上形式化检查,都能用已知的成本换更紧的界。其二,前沿审查臂说明:付得起按条计费前沿模型的机构,眼下确实能在经验上追平执行门的准确率——但换不来确定性、可审计性,也防不了模型更新带来的漂移;跨域检验(§5.6)把这一点磨成了一条边界——在教科书熟悉的任务上前沿审查器几乎追平执行,所以门的**检测**红利(区别于成本与确定性红利)最大处,恰在 HDL 这类专门、训练欠代表的领域。其三,这套方法只适用于成果可被机器执行检验的领域;它对作文批改无话可说。

## 7. 局限与未来工作

第一,这是技术性、对抗性的评测,不是课堂研究:本文对学习效果、信任校准、可用性不做任何断言。顺理成章的下一步,是教师盲评(门控流与未门控流对比),再往后是课程试点;§3.5 的可测指标就是为这个迁移预备的。第二,金标是构造出来的,不是采集来的:构造法给了精确标签和可控覆盖,但对真实学生错误的长尾覆盖不足;在真实提交日志上的复现已列入计划(包括 L0 能轻松处理、学生却觉得费解的编译类错误)。第三,关于向主语料之外的迁移,如今我们有的是实证而不只是论证:一套独立编写、横跨十四类设计的 130 题 HDL 语料复现了全部结论(§5.5),入门 Python 上的跨域检验则表明机制可迁移、同时划出了它红利的边界(§5.6,那里前沿审查器在教科书任务上几乎追平执行);至于仿真语义之外的形式化学科——如需模型检查而非差分测试的 CDC 类——仍是未来工作。第四,脚手架只测了 12 个题目、一种生成器配置;提示语的质量(有别于检查点的有效性)未测,需要专家评分。第五,LLM 各臂只跑了一轮:前沿审查器两遍 κ = 1.0 缓解了、但没消除运行间方差的疑虑;执行门是确定性的,没有这个问题。

## 8. 结论

已部署的 LLM 导师,正以多项已发表审计所记录、且形成性反馈理论认定"有害"的比率输出错误反馈(Kazemitabaar et al., 2024; Azaiz et al., 2024; Yasir et al., 2026b)——而现行的补救,要么会失效,要么不可扩展,要么继承了它本要治的病。本文证明:在成果可执行的学科里,这个问题有工程解。一个框架——执行验证器掌裁决,LLM 掌表达;一个参考系统——96% 的评判带验证器背书、准确率 99.5%,对抗注入的有害反馈 100% 被拦,单次 71 毫秒、成本近零,交付的每个练习步骤可证通向正确解。对实践分量最重的,是同题对决那一组数字:执行门追平前沿模型审查,普通模型审查两头失守——所以真正的选择题是:**要按条付费买来的安全,还是要长在架构里的安全?** 这些结论在一套独立编写、横跨十四类设计的公开语料上原样复现,并在入门 Python 上跨域成立(同时诚实地界定了:在训练高度代表的任务上,前沿审查器就够用,门的检测优势最大处在 HDL 这类专门领域)。对久缺可信工具、又肩负战略人才使命的硬件设计教育,以及正在寻找"含可证组件的混合架构"的教育 AI 研究,验证门控辅导给出了一个保证可查、过程可审、边际免费的模板。
